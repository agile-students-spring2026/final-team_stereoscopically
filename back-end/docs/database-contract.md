# Database Contract

## Purpose

This document defines the database contract for the back-end code in `back-end/`.

It covers:
- database connection and storage primitives
- collections and document shapes
- ID/reference rules between documents and API payloads
- retention and cleanup behavior
- index requirements

---

## Runtime database model

- Database provider: MongoDB Atlas
- ODM/client: Mongoose + MongoDB Node driver
- Default database name: `stereoscopically` (overridable by `MONGODB_DB_NAME`)
- Binary storage: GridFS bucket `media`

GridFS collections used by the bucket:
- `media.files`
- `media.chunks`

Application collection used for user drafts/exports:
- `creations`

---

## Collection contracts

### `creations`

Source of truth:
- `src/models/creationModel.js`

Current schema fields:
- `_id: ObjectId` (generated by MongoDB)
- `userId: ObjectId | null` (references `users` when draft is tied to an account; null/absent for guest-only drafts)
- `ownerKey: string` (trimmed when present; used for guest drafts when `userId` is not set; omitted for account-bound drafts)
- `title: string` (required, trimmed, max length 200)
- `status: 'draft' | 'exported'` (required, defaults to `draft`)
- `editorPayload: mixed` (required)
- `exportAssetId: string | null` (optional; media ID reference as string)
- `trackedMediaIds: string[]` (default `[]`; media IDs accumulated for cleanup on delete)
- `createdAt: Date` (auto via timestamps)
- `updatedAt: Date` (auto via timestamps)

**Ownership rule**

- Each document uses one persistence mode after create via API today: either **account-bound** (`userId` set on create) **or guest** (`ownerKey` stored while `userId` remains unset). Treat `ownerKey` as loose browser-scope scoping only, not a bearer secret.

**HTTP semantics** for list/read/write/delete are documented in **`back-end/readme.md`** (â€śCreations (drafts / exports)â€ť).

### `media.files` (GridFS metadata)

Source of truth:
- `src/services/mediaStore.js`
- `src/services/mediaService.js`

Core GridFS fields (managed by MongoDB/GridFS):
- `_id: ObjectId`
- `length: number`
- `chunkSize: number`
- `uploadDate: Date`
- `filename: string`
- `contentType: string`
- `metadata: object`

Current app metadata conventions:
- `metadata.mimeType: string` (always set by media store write path)
- `metadata.kind: 'image' | 'video' | 'gif'`
- `metadata.operation: string` (examples: `upload`, `crop`, `adjust`, `preset-filter`, `text-overlay`, `trim-video`, `preset-video-filter`, `export`)
- `metadata.sourceMediaId: string` (when output is derived from a prior media record)
- operation-specific fields where applicable, e.g.:
  - image export: `width`, `height`, `letterboxColor`
  - image adjust: `adjustments`
  - image preset filter: `preset`
  - gif trim: `resizePreset`, `resizeBorderColor`, `textOverlay`

### `media.chunks` (GridFS binary chunks)

Core GridFS fields:
- `_id: ObjectId`
- `files_id: ObjectId` (references `media.files._id`)
- `n: number` (chunk order)
- `data: Binary`

## `editorPayload` contract (persisted in `creations`)

`editorPayload` is currently a mixed object but follows stable shapes from frontend payload builders.

### Image payload (current)

- `kind: 'image'`
- `version: number` (current `2`)
- `sourceMediaId: string | null`
- `workingMediaId: string | null`
- `backendMediaId: string | null` (legacy alias; mirrors `workingMediaId`)
- `lastCropBoxPx: object | null`
- `colorAdjustments: object | null`
- `selectedImageFilterPreset: string`
- `selectedPreset: string | null`
- `letterboxColor: string`

### Video payload (current)

- `kind: 'video'`
- `version: number` (current `2`)
- `sourceMediaId: string | null`
- `workingMediaId: string | null`
- `backendMediaId: string | null` (legacy alias; mirrors `workingMediaId`)
- `trimRange: { start: number, end: number }`
- `resizePreset: string | null`
- `resizeBorderColor: string | null`
- `selectedSpeedPlaybackRate: number`
- `textOverlaySettings: object | null`
- `selectedFilterPreset: string`

Notes:
- Back-end create/update endpoints validate `editorPayload` as an object, but do not yet enforce full nested shape validation.
- `backendMediaId` is retained for backward compatibility during payload migration.

## ID and reference rules

- `creations._id` is the canonical creation identifier.
- Media API IDs are serialized ObjectIds from GridFS (`media.files._id`).
- `creations.exportAssetId` should reference a GridFS media ID string when a creation is exported.
- `editorPayload.sourceMediaId` and `editorPayload.workingMediaId` should reference GridFS media IDs when present.
- API and frontend treat media IDs as strings; media store converts/validates against ObjectId for reads/deletes.

---

## Lifecycle and retention rules

Current behavior:
- Media binaries are persisted in GridFS until explicitly deleted.
- Draft/export records in `creations` are persisted until explicitly deleted.
- No automatic TTL-based purge is currently wired into active storage flows.

Compatibility note:
- `MEDIA_TTL_MS` and `MEDIA_PURGE_INTERVAL_MS` remain in `src/config/constants.js` as legacy constants.

Operational implication:
- Storage growth should be monitored until an explicit cleanup policy/job is implemented.

## Index requirements

### Required indexes

`creations`:
- compound index `{ userId: 1, updatedAt: -1 }`
- compound index `{ ownerKey: 1, updatedAt: -1 }`

`media.files` / `media.chunks`:
- GridFS-required indexes managed by MongoDB/GridFS driver semantics
  - `media.files`: index on `filename` + `uploadDate`
  - `media.chunks`: unique compound index on `files_id` + `n`

### Optional future indexes

Add only after query-profile evidence:
- `creations.userId` sparse/partial helpers (only if profiling shows inefficient guest vs user scans)
- `creations.status` (if status-filtered queries become frequent)
- `creations.exportAssetId` (if reverse lookup by exported media becomes frequent)

## Validation ownership

Back-end validation currently enforces (creations REST):
- **Signed-in**: `ownerKey` must not appear as non-empty/non-whitespace in create body.
- **Guest**: `ownerKey` required on create body; **`ownerKey`** query param required for `GET`/`PATCH`/`DELETE` `/api/creations/:id` unless JWT identifies the owning user.
- `title`: required, trimmed, max length 200.
- status enum (`draft`/`exported`).
- `editorPayload`: object presence.
- **`GET /api/creations` for guests**: non-empty query `ownerKey`.

Back-end does not yet enforce:
- strict versioned schema validation for nested `editorPayload` by `kind`

Recommendation:
- introduce explicit payload validators at controller boundary if payload compatibility issues increase.

## Change management rules

When changing persistence shape or references:
- update this document in the same PR
- include backward-compatibility handling (or migration plan)
- keep API error shapes stable
